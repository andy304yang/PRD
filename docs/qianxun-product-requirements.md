# 千寻产品需求基准文档

本文档是千寻 Demo 的产品口径基准。后续遇到产品相关问题，优先从本文档判断；若与代码现状冲突，以本文档作为产品意图，以代码审计结果作为实现现状。

## 产品基本信息

- 产品名称：千寻
- 公司名：屿智同行
- 英文品牌标识：QIANXUN AGENT WORK
- Slogan：不是投简历，而是让你的分身 Agent 先去工作。
- 产品形态：微信小程序形态优先，用 H5 / React 模拟小程序体验。

## 核心定位

千寻不是传统招聘 App，也不是简历投递平台，而是“人 & Agent 协同工作时代的人才基础设施”。

产品主张：

- 过去招聘看学历、履历、面试表现；未来招聘更看候选人与 AI Agent 协同后的真实交付能力。
- 候选人创建自己的“能力分身 Agent”，不是简单上传简历。
- 分身 Agent 基于求职方向和 Skills，进入真实岗位任务中试跑、拆解、执行、纠错、产出，并留下过程证据。
- 用人方不只看最终作品，而是看候选人如何拆任务、写 Prompt、编排工作流、验证 AI 输出、转成业务交付。

MVP 场景：

- 首发锚定“实习生 / 0-3 年经验年轻人”的求职与用人匹配。
- 包含游客模式、C 端求职者视角、B 端用人方视角。
- 主要方向：AI 产品、AI 运营、AI 增长、AI 市场研究、AI 销售、AI 内容、AI 产品运营、新媒体运营、商业策略、数据分析、UX / AI 原型设计。

重点表达：

- 不是投简历，而是派出分身先去工作。
- 不是看最终作品，而是看过程证据。
- 不是传统笔试，而是 Co-Task Challenge。
- 不是能力标签，而是可执行的 Skills。
- 不是候选人单点表现，而是人 & Agent 协同交付能力。
- C 端掌握开放权，B 端只能在授权后查看完整过程与结果。

## 全局技术与 Demo 规则

- 使用 React + TypeScript + H5 模拟微信小程序。
- 页面最大宽度 430px，桌面端居中展示。
- 必须模拟微信小程序状态栏、顶部导航、胶囊按钮、底部 TabBar、弹窗、Toast、授权登录、页面返回、列表卡片、动效执行过程。
- 默认首次进入是游客模式。
- 本地状态和 localStorage 可模拟账号、角色、分身、Skills、任务、Run 版本、消息、B 端激活状态、激活码状态。
- 不需要真实后端、真实微信登录、真实支付、真实身份证认证、真实人脸识别、真实 AI API、真实 PDF 后端、真实短信、真实企业工商接口。
- 微信授权、AI 执行、PDF 预览、任务推荐、企业检索、激活码、短信通知全部用 Mock 数据 + 动效模拟。
- 非一级页面左上角必须有返回按钮，返回到进入该页面前的操作页面。
- 展开的弹窗统一支持点击非弹窗区域关闭。

全平台删除项：

- 不展示 token / tokens / Token 消耗 / 预计 Token。
- 不展示评分 / 质量评分 / 平均质量评分。
- 不做人脸认证。
- 不做真实招聘投递。

## 视觉风格

关键词：AI Native、未来感、可信、年轻、职业化、轻量、流畅、不是传统招聘软件。

视觉要求：

- 主色：黑 / 白 / 深灰。
- 辅助色：电光蓝、青绿色、淡紫渐变。
- 卡片：圆角 20px、轻微阴影、玻璃拟态、细边框。
- 英文小标题可使用：AGENT WORK、CO-TASK CHALLENGE、SKILL GRAPH、RUN EVIDENCE、AGENT VERSION。
- 任务执行时使用流动线条、步骤高亮、进度条、节点跳动、日志逐条浮现。
- 空状态可使用呼吸动效、半透明 Agent 轮廓、光晕、虚线卡片。

## 游客模式与登录

默认游客按 C 端用户视角展示，保留 5 个 Tab：

- 首页
- 机会市场
- 我的分身
- 任务视界
- 消息

游客规则：

- 可以查看内容。
- 不能执行关键操作。
- 创建分身、派出分身、批量派出、查看真实过程、开放任务结果、发送消息等操作必须弹登录 / 注册引导。

未登录关键操作拦截弹窗固定文案：

- 标题：登录后，让你的分身 Agent 先去工作
- 正文：创建分身、派出任务、查看过程证据都需要账号状态。
- 按钮：登录 / 注册、稍后再说

角色选择弹窗：

- CANDIDATE / 我是求职者：创建你的能力分身 Agent，让它先进入真实岗位任务中试跑、产出和证明你。
- EMPLOYER / 我是用人方：发布模拟实战任务，查看候选人分身的过程证据与交付结果。
- 点击角色后进入“微信一键授权”确认弹窗，不直接登录。

授权确认弹窗：

- 标题：微信一键授权
- 求职者正文：你已选择“我是求职者”。Demo 将模拟微信授权并创建本地账号。
- 用人方正文：你已选择“我是用人方”。Demo 将模拟微信授权并创建本地账号。
- 按钮：允许并进入、取消
- 点击允许后 Toast：授权成功。

授权状态重置：

- 选择求职者：重置 hasAgent、skills、runs，进入 C 端。
- 选择用人方：重置 activated，微信名默认填写 B 端姓名，进入 B 端未激活状态。

设置弹窗：

- 展示当前体验身份：游客 / 求职者 / 用人方。
- 文案：你可以清空本地 Demo 状态，回到游客模式，或重新选择体验身份。
- 按钮：重新选择角色、注销账号、关闭。
- 注销账号清空 localStorage 中账号、角色、分身、Skills、Run、消息、B 端激活状态等 Demo 状态，回到游客模式，Toast：账号已注销。

## C 端：首页

页面定位：求职者总览入口，强调“不是投简历，而是让分身 Agent 先去工作”。

顶部区域：

- 品牌标识：QIANXUN AGENT WORK
- 问候语：哈喽，Xiao
- 不展示分身状态、token、评分、姓名右侧 agent work 标识。

主宣传区：

- 主标题：千寻，让你的分身先去工作
- 副标题：上传简历或说一段自我介绍，生成你的能力 Agent，让它进入真实岗位任务中试跑、产出和证明你。
- 按钮：创建 / 更新我的分身、进入任务视界。
- 点击分别跳到“我的分身”和“任务视界”Tab。

未创建分身派出拦截：

- 已登录但未创建分身时，可浏览首页和任务，但不能派出分身。
- 弹窗标题：先创建你的分身 Agent
- 正文：派出任务前，需要先生成你的能力分身。上传简历或和 AI 简单聊几句即可创建。
- 按钮：去创建分身、稍后再说。

我的分身摘要卡：

- 分身名称：Xiao AI
- 求职方向：产品 > AI 产品策划
- Skills 数量：18
- 已结束任务：100
- 已开放版本：10
- 按钮：查看分身
- 说明：Skills 数量 18 与 Skills 模块默认 4 个 Skills 示例属于 Demo 展示数据，不作为严格功能逻辑。
- 不展示分身状态、token、评分。

平台数据宣传卡：

- 标题：千寻正在形成 AI-Native 人才基础设施
- 数据：10 行业、100 公司、1000 岗位任务、102,483 用户分身
- 这是产品宣传数据，不要求来自运行态 API。

推荐任务：

- 标题：RECOMMENDED 推荐任务
- 右上角：查看全部，点击跳到机会市场。
- 首页展示 3 个任务卡。
- 首页 3 条任务必须来自后台任务池中与用户匹配档位为“极高”的任务。
- 用户进入或退出后重新进入时，首页 3 条与机会市场 18 条同时刷新。
- 刷新时从极高匹配任务池抽 9 条：3 条首页，6 条机会市场。
- 机会市场其余 12 条为高匹配 6 条、中匹配 6 条。
- 首页 3 条在用户退出并重新进入前保持不变。
- 机会市场换一批不能与当前首页 3 条重复。
- 首页任务即使已派出，也仍展示在首页，直到退出并重新进入。

首页任务卡字段顺序：

1. 任务名称
2. 匹配分类
3. 一级分类 · 二级分类 · 岗位名称
4. 公司名
5. 行业
6. 任务描述，两行省略

首页派出逻辑：

- 未登录：弹登录 / 注册引导。
- 已登录但未创建分身：弹创建分身引导。
- 已登录且已创建分身：弹二次确认，不直接执行。
- 确认后 Toast：分身已派出；任务进入任务视界“执行中”；首页任务不消失；按钮变为“已派出”。
- 同一任务已经 queued 或 running 时，不重复创建 Run，Toast：分身已派出，任务进入任务视界。

## C 端：我的分身

页面定位：创建、查看、维护求职者能力分身 Agent。

顶部：

- 标题：我的分身
- 副标题：维护你的求职方向与 Skills，决定分身 Agent 如何执行任务

未创建状态：

- 标题：你还没有创建分身 Agent
- 副标题：上传简历或和 AI 简单聊几句，生成你的初版能力分身。
- 中央两个卡片按钮：上传简历、AI 对话生成。
- 视觉：呼吸动效 Agent 轮廓，淡紫 / 电光蓝光晕，入口卡片居中。

上传简历：

- 点击后弹模拟上传弹窗。
- 上传成功后进入独立“分身生成中”居中 Modal，不用飘条。
- 流程文案：正在读取简历关键信息、正在识别教育背景与项目经历、正在提取目标岗位方向、正在生成能力框架、正在沉淀初始 Skills、初版分身生成完成。
- 过程约 2 秒。
- 成功后展示已创建分身状态。
- 失败展示后端 Mock message，例如：生成失败：简历内容过少，请补充项目经历后重试。

AI 对话生成：

- 进入模拟聊天界面或全屏弹层。
- Bot 开场：你好，我会用 2 分钟帮你生成初版能力分身。请先用一段话介绍你的教育背景、经历、想找的岗位和你擅长用 AI 做什么。
- Bot 追问 2-3 个问题：目标岗位方向、最接近项目、常用 AI 工具。
- 点击生成后使用与上传一致的生成弹窗。

Skill 生成规则：

- 首次创建分身自动生成 2 个初始 Skills。
- 采用“求职方向映射模板 + 用户经历轻度填充”。
- 求职方向是唯一必须确认字段。
- 提取不到专业、实习、岗位相关信息时生成失败。
- 初始 Skill 1：岗位能力框架类 Skill。
- 初始 Skill 2：具体求职内容类 Skill。

已创建状态：

- 不再展示上传简历和 AI 对话生成入口。
- 基本信息可编辑：姓名、学校、学历、工作 / 项目经历、求职方向。
- 不展示实名认证 / 人脸认证。
- 不增加额外保存按钮，点击空白处自动保存，Toast：信息已保存。

求职方向：

- 一级分类：产品、运营、市场、销售、研究、设计、内容、增长、数据分析、商业策略。
- 同一时间只能选择一个求职方向。
- 更改方向时弹窗提醒：求职方向会影响分身 Agent 的任务匹配与执行方式。建议同步检查并更新 Skills。
- 按钮：稍后再说、去检查 Skills。

Skills 模块：

- 标题：Skills｜分身的工作方法
- 副标题：Skill 不是能力标签，而是你的 Agent 可复用的工作流程、Prompt 策略和质检标准。
- 最多 20 个。
- 卡片包含：标题、启用状态、适用任务类型、版本、来源、用途、右上角启停开关、底部编辑 / 复制 / 删除。
- 启用 Toast：Skill 已启用，后续任务可调用。
- 停用 Toast：Skill 已停用，后续任务将不再调用。
- 默认展示 4 个 Skills 示例，仅作 Demo 展示。
- 新增满 20 个时按钮显示“已满”，提示：当前最多维护 20 个 Skills。
- 复制后标题为“原标题 副本”，来源“手动创建”，版本 v1.0。
- 删除前二次确认，确认后 Toast：Skill 已删除。
- Skill 在任务派出时封存为快照，后续删除 / 编辑 / 停用不影响已结束、已开放、执行中任务。

## C 端：机会市场

页面定位：岗位任务市场，不是职位列表。

顶部：

- 标题：机会市场
- 副标题：选择真实岗位任务，让你的分身先去试跑
- 删除顶部搜索框、一级 / 二级分类陈列和筛选。
- 保留任务列表、换一批、多选模式、全选本屏、批量派出。

推荐规则：

- 每次进入或退出后重新进入时，首页 3 条和市场 18 条同时刷新。
- 市场 18 条：极高匹配 6 条、高匹配 6 条、中匹配 6 条。
- 市场极高匹配 6 条不得与首页当前 3 条重复。
- 已派出、执行中、已结束、已开放、已下线任务不再出现在机会市场。
- 派出后列表自动补位，优先保持 18 条。
- 换一批 Toast：已换一批任务，并清空多选状态。

任务卡：

- 展示顺序与首页任务卡一致。
- 额外展示匹配解释：命中你的 3 个 Skills：...
- 按钮：查看详情、派出分身。
- 卡片非按钮区域也进入任务详情。
- 不展示 token、评分、数字化匹配分数。

任务详情：

- 独立界面，不是弹窗。
- 字段：任务标题、发布方、行业、一级分类、二级分类、岗位名称、任务内容、匹配档位、匹配解释、当前 Agent 运行中数量、隐私与开放说明。
- 隐私说明：分身执行过程和结果默认仅自己可见。任务完成后，你可以选择是否开放给用人方查看。

派出确认：

- 单任务派出必须二次确认。
- 标题：确认派出 Xiao AI？
- 正文：分身将基于你当前启用的 Skills 自动拆解并执行任务。过程和结果默认仅自己可见，完成后你可以选择是否开放给用人方。
- 批量派出也必须二次确认。
- 同一任务 queued 或 running 不重复创建 Run。

## C 端：任务视界

页面定位：分身 Agent 工作台，展示执行过程、过程证据、结果产出、版本状态和开放状态。

顶部：

- 标题：任务视界
- 副标题：查看你的分身如何拆解、执行、纠错和交付

状态 Tab：

- 执行中
- 未开放
- 已开放
- 全部

规则：

- “未开放”展示已结束但未开放的 Run 版本，也包括被终止版本。
- 已开放是版本维度，不是任务整体维度。
- 按 Run 版本状态筛选，展示时按任务聚合。
- 同一个任务在同一个 Tab 下只展示一张任务卡。
- 同一个任务如果多个版本处于不同状态，可以同时出现在多个 Tab。
- 同一个 Run 版本只能属于一个状态。
- 任务卡不展示 Skill 快照、token、评分。
- 任务卡上的 Run 版本只展示，不可点击。
- 具体版本选择只发生在查看过程、预览结果、开放给用人方、分享卡、迭代 Skills。

执行过程页：

- 必须有动态流程图。
- 节点：读取任务、匹配 Skills、拆解目标、生成执行计划、多轮执行、质检纠错、生成交付物、归档过程证据。
- 展示状态、简短说明、模拟日志。
- 动效：当前节点发光、连线流动、日志逐条浮现、进度条推进。

终止任务：

- 执行中任务终止前二次确认。
- 终止后版本进入“未开放”，可查看已产生过程记录，不展示完整最终结果。

未开放版本：

- 正常完成按钮：查看过程、预览结果、开放给用人方、分享卡、迭代 Skills、再跑一版。
- 预览结果包含标题、候选人分身、求职方向、任务信息、执行摘要、分析过程、最终方案、自检与修正记录、运行时间、二维码占位。

开放给用人方：

- 一次只能选择一个版本开放。
- 可针对同一任务开放多个版本。
- 开放后 Toast：已开放给用人方。
- 被选择版本进入“已开放”，从“未开放”移除。
- 每次开放都会进入与对应用人方的同一个消息对话框。
- B 端对应任务下候选人的对应版本状态变为已开放。

分享卡：

- 包含千寻 Logo、“我的分身完成了一个真实岗位任务”、分身名称、求职方向、任务标题、已调用 Skills 数量、二维码区域。
- 保存图片 Toast：分享卡已生成，Demo 环境中为模拟保存。

再跑一版：

- 同一个任务允许多次执行，每次创建新 Run 版本。
- 每个版本保存执行时间、Skill 快照、过程日志、结果报告、开放状态、当前状态。
- 可提供简单版本对比。

## C 端：消息

- 顶部标题：消息。
- 取消顶部子 Tab，所有消息平铺。
- 系统消息置顶为一个对话框。
- 系统消息只通知任务完成。
- 暂无消息空状态按钮：去机会市场看看。
- 消息卡片：最新消息与时间同一行，消息左、时间右，消息一行省略。
- 日期是当天显示 hh:mm，否则显示 YYYY-MM-DD。
- 点击消息卡片进入双方沟通页面，不使用弹窗。
- 沟通方姓名下展示“公司 · 行业”。
- 支持发送文本消息，Toast：消息已发送，并同步写入 B 端。

C 端主动开放消息卡：

- 文案：你已向 NoteFlow AI 开放「任务标题」Run v2 的完整过程与结果。
- 默认消息：你好，我已向你开放「{任务标题}」{Run 版本} 的完整过程与结果，欢迎查看；如需我进一步说明，我可以在这里补充。
- 去掉“查看任务”“回复”“已处理”。

## 状态同步

- C 端开放任务版本后，B 端对应任务下候选人对应版本状态变为已开放，B 端消息出现候选人开放消息。
- B 端邀请开放后，C 端消息中出现邀请，进入双方唯一对话框。
- C 端同意开放后，该任务版本进入 C 端已开放，B 端收到同意通知，B 端可查看对应版本完整内容。
- C 端拒绝开放后，B 端收到拒绝通知，双方仍可继续发消息。
- 任务下线后，C 端机会市场不再展示该任务，C 端任务视界已有任务不受影响，已完成候选人仍可选择开放给该用人方。
- B 端激活后，企业信息从申请信息带回，系统消息出现欢迎消息，所有操作权限解锁。

## 任务执行动效与并发

- 派出分身后任务进入执行中。
- 用 setInterval 模拟 0% 到 100%。
- 阶段推进时更新流程节点、日志、当前阶段、进度条。
- 完成后自动进入未开放状态。
- 生成模拟结果报告和系统消息：你的分身 Xiao AI 已完成任务「xxx」。
- 内部最多 3 个任务推进进度。
- 超过 3 个保持执行中，但显示：正在准备执行环境。
- 不向用户暴露“最多并发 3 个”。
